# Naroom API 저장소 작업 지침

## 지침 구조

Claude Code는 다음 저장소 지침을 불러온다.

- `CLAUDE.md`: 백엔드 프로젝트 및 작업 규칙
- `.claude/CLAUDE.md`: OMC 운영 규칙
- `docs/PRODUCT_CONTEXT.md`: 승인된 Naroom 제품 맥락

현재 이 저장소에는 `AGENTS.md`가 없다.
나중에 해당 파일을 의도적으로 생성하기 전에는 `@AGENTS.md` import를 추가하지 않는다.

## 프로젝트 개요

- 프로젝트: `naroom-api`
- 역할: Naroom 모바일 앱을 위한 백엔드 API
- 기본 패키지: `com.naroom.api`
- 일반 개발 브랜치: `dev`
- 안정 브랜치: `main`
- 제품 맥락: `docs/PRODUCT_CONTEXT.md`

Naroom은 개인 기록을 회고, 패턴 발견, 사용자가 직접 선택하는 작은 변화와 연결하는 장기 자기이해 플랫폼이다.

제품 동작을 구현하기 전에 `docs/PRODUCT_CONTEXT.md`와 현재 GitHub Issue 또는 사용자가 제공한 요구사항을 확인한다.

작업 범위와 완료 기준은 다음 순서로 판단한다.

1. 사용자의 현재 명시적 지시
2. 현재 GitHub Issue 또는 작업 요구사항
3. 승인된 제품 문서, ADR, API 계약, 마이그레이션
4. 기존 코드와 테스트
5. 이 저장소 공통 지침

동작, 스키마, 보안, 공개 API 호환성에 영향을 주는 충돌이 있으면 임의로 선택하지 않는다. 충돌을 보고하고 결정을 요청한다.

## 현재 기술 기준

- Java 21
- Spring Boot 4.1.0
- Gradle 9.5.1
- Gradle Kotlin DSL
- Gradle Wrapper
- Spring Web
- Bean Validation
- Spring Boot Actuator
- Spring Data JPA
- Supabase PostgreSQL
- Flyway
- JUnit 5
- Spring Boot Test

실제로 설치된 의존성은 `build.gradle.kts`, Gradle 상태, 설정 파일, 기존 코드를 기준으로 판단한다.

Spring Security, Springdoc OpenAPI, 카카오 연동, Redis, 외부 API 등은 프로젝트에 실제로 추가되기 전까지 설치된 것으로 간주하지 않는다.

현재 작업에 필요하지 않은 의존성, 플러그인, 인프라, 외부 연동은 임의로 추가하거나 업그레이드하지 않는다.

## 주요 경로

- 애플리케이션 코드: `src/main/java/com/naroom/api`
- 테스트 코드: `src/test/java/com/naroom/api`
- 설정: `src/main/resources`
- Flyway 마이그레이션: `src/main/resources/db/migration`
- 제품 맥락: `docs/PRODUCT_CONTEXT.md`
- API 계약 초안: `docs/contracts/drafts`
- 생성된 OpenAPI 스냅샷: `docs/openapi`
- 아키텍처 문서: `docs/architecture`
- 도메인 문서: `docs/domain`
- ADR: `docs/adr`
- 앱 상태 API: `GET /api/v1/health`
- Actuator 상태 API: `GET /actuator/health`

이미 같은 목적의 승인된 문서 경로가 있으면 중복 디렉터리를 생성하지 않는다.

## 로컬 실행

```bash
./gradlew bootRun
```

기본 로컬 프로필은 `local`이다.

애플리케이션 실행 중 `.env.local`이 로드될 수 있다. Claude Code는 승인된 실행 명령을 사용할 수 있지만 `.env.local` 내용을 읽거나 출력하거나 요약하거나 수정하거나 노출해서는 안 된다.

## 빌드 및 검증

Gradle Wrapper를 사용한다.

```bash
./gradlew test
./gradlew clean build
```

코드를 변경한 후 다음을 수행한다.

1. 변경 동작에 대한 집중 테스트를 실행한다.
2. `./gradlew test`를 실행한다.
3. `./gradlew clean build`를 실행한다.
4. 공개 API 계약이 변경되면 OpenAPI를 재생성하고 검증한다.
5. 최종 diff를 검토한다.
6. 변경 파일, 실행 명령, 결과를 보고한다.
7. 실행하지 못했거나 실패한 검증과 이유를 보고한다.

필수 검증이 실패했다면 완료됐다고 보고하지 않는다.

## 작업 범위 관리

수정 전에 다음을 수행한다.

1. 현재 Issue 또는 사용자 요구사항을 확인한다.
2. 관련 제품 문서, 코드, 테스트, 마이그레이션, API 계약을 확인한다.
3. 포함 범위와 제외 범위를 구분한다.
4. 수정 예정 파일과 검증 방법을 간단히 알린다.

사용자의 요청이 명확하고 승인 범위 안이라면 별도의 재승인을 기다릴 필요는 없다.

다음 상황에서는 작업을 중단하고 확인한다.

- 제품 문서가 실질적으로 충돌하는 경우
- 공개 API 결정이 확정되지 않은 경우
- 적용된 마이그레이션과 요청된 엔티티 모델이 충돌하는 경우
- 보안 관련 정책이 결정되지 않은 경우
- 승인되지 않은 외부 서비스나 비밀값이 필요한 경우
- 요청 범위를 크게 확장해야 하는 경우

요청과 관계없는 리팩터링, 이름 변경, 정리, 의존성 업그레이드, 아키텍처 재설계를 수행하지 않는다.

## 패키지 및 아키텍처 규칙

- 비즈니스 API는 `/api/v1` 하위에 작성한다.
- Actuator와 운영 관리 엔드포인트는 예외다.
- `com.naroom.api` 하위에서 기능 중심 패키지 구조를 사용한다.
- `global`에는 실제로 여러 기능에서 공유되는 횡단 관심사만 둔다.
- 기능 전용 로직을 일반적인 공통 또는 유틸리티 패키지에 넣지 않는다.
- 생성자 주입을 우선한다.
- 요청값 검증에는 Bean Validation을 우선 사용한다.
- Controller는 HTTP 처리에 집중한다.
- Controller에 비즈니스 로직을 넣지 않는다.
- 트랜잭션 경계는 Service 또는 Application 계층에 둔다.
- 요청 DTO, 응답 DTO, 도메인 모델, 영속성 Entity를 분리한다.
- JPA Entity를 Controller 응답으로 직접 노출하지 않는다.
- 적절한 불변 요청·응답 모델에는 Java record를 사용할 수 있다.
- 클래스, 메서드, 필드, 변수는 역할이 드러나는 영어 이름을 사용한다.
- 기존 코드 스타일과 확정된 프로젝트 구조를 따른다.
- 공개 요청·응답 계약을 바꿀 때는 영향과 이유를 설명한다.

## API 계약 및 문서 규칙

공개 API 계약에는 다음이 포함된다.

- 엔드포인트 경로와 HTTP 메서드
- 인증 요구사항
- 요청·응답 스키마
- 필수, 선택, nullable 필드
- 검증 제약
- Enum 값
- 오류 코드와 오류 스키마
- 날짜, 시간, 시간대 형식
- 페이지네이션 규칙
- 호환성과 폐기 정책

공개 API는 다음 순서로 작업한다.

1. API 계약을 확정하거나 초안을 작성한다.
2. Controller, DTO, 검증, 동작을 구현한다.
3. 집중 테스트와 통합 테스트를 추가한다.
4. OpenAPI 문서를 생성한다.
5. 생성된 문서를 검증한다.
6. 문서 자동화 정책에 따라 승인된 스냅샷을 커밋 대상으로 관리한다.

규칙은 다음과 같다.

- OpenAPI는 실행 가능한 HTTP 계약을 담당한다.
- Markdown은 개념, 정책, 흐름, 결정을 설명한다.
- ADR은 중요한 아키텍처 결정과 이유를 보존한다.
- Flyway는 DB 스키마 이력을 담당한다.
- 생성된 OpenAPI를 수동으로 수정하지 않는다.
- 일반 Markdown 문서에 상세 엔드포인트 스키마를 중복 작성하지 않는다.
- 코드, 타입, 검증만으로 계약이 충분히 표현되지 않을 때만 OpenAPI 어노테이션을 추가한다.
- 중요한 설계 결과를 터미널 출력이나 `.omc/`에만 남기지 않는다.

## 모바일 API 호환성

백엔드와 모바일 앱은 서로 다른 시점에 배포될 수 있다.

- 추가형이며 하위 호환되는 변경을 우선한다.
- 기존 필드를 즉시 삭제하거나 이름을 변경하지 않는다.
- 기존 필드의 의미를 바꾸지 않는다.
- 호환성이 필요하면 신규 필드를 선택값으로 설계한다.
- Enum을 추가할 때 앱 fallback 동작을 고려한다.
- 호환되지 않는 변경은 명시적인 호환 또는 API 버전 계획을 세운다.
- 앱이 의존하기 전에 호환 가능한 백엔드를 먼저 배포한다.
- 지원 앱 버전이 더 이상 사용하지 않을 때만 구버전 동작을 제거한다.

## 성공 응답 및 오류 규칙

- 문서화된 하나의 성공 응답 규칙을 사용한다.
- HTTP 204처럼 본문이 없어야 하는 응답은 래핑하지 않는다.
- 공개 오류 응답은 하나의 문서화된 형식을 사용한다.
- 공개 오류에는 안정적인 기계 판독용 오류 코드가 있어야 한다.
- 검증 오류는 잘못된 필드를 표시할 수 있지만 입력된 비밀값을 노출하면 안 된다.
- 인증·인가 실패도 Controller 예외와 동일한 공개 오류 계약을 사용한다.
- 스택 트레이스, SQL, 내부 클래스명, 내부 경로, 공급자 원문 응답, 토큰, 민감한 사용자 데이터를 노출하지 않는다.
- 앱이 사용하는 공개 오류 코드는 OpenAPI 또는 승인된 API 문서에 포함한다.

아직 성공 래퍼나 오류 스키마가 승인되지 않았다면 임의로 확정하지 않는다. 대안을 작성하고 결정을 요청한다.

## 데이터베이스 및 Flyway 규칙

- 이미 적용된 Flyway 마이그레이션은 불변이다.
- 적용된 마이그레이션을 수정, 이름 변경, 순서 변경, 삭제하지 않는다.
- 모든 스키마 변경은 새 마이그레이션으로 추가한다.
- Entity는 실제 적용된 DB 스키마에 맞춘다.
- Entity 구현을 편하게 만들기 위해 스키마를 바꾸지 않는다.
- 고유 제약, 외래키, 인덱스, 소유권, 타임스탬프, 상태 전환, 삭제 방식을 확인한다.
- Repository는 영속성 책임만 가진다.
- 영속성 Entity를 공개 DTO로 노출하지 않는다.
- `spring.jpa.hibernate.ddl-auto`는 Flyway 기반 validate 정책과 호환되게 유지한다.
- 승인된 UTC 및 ISO 8601 정책에 따라 시간을 저장하고 교환한다.

마이그레이션, Entity, 데이터 사전, 제품 문서가 일치하지 않으면 구현을 중단하고 차이를 보고한다.

## 계정 및 식별자 규칙

- Naroom 회원은 Naroom 내부 회원 ID로 식별한다.
- `Member`와 `SocialIdentity`를 별도 개념으로 유지한다.
- 외부 공급자 식별자는 회원과 분리해 저장한다.
- 카카오 식별자는 카카오 서비스 사용자 ID를 안정적인 식별자로 사용한다.
- 공급자와 공급자 사용자 ID에 대해 승인된 고유 제약을 적용한다.
- 이메일, 닉네임, 이름, 생년월일, 성별, 연령대는 기본 식별자가 아니다.
- 개인정보가 같다는 이유로 계정을 자동 통합하지 않는다.
- Beta 1에서는 카카오 로그인만 구현한다.
- 데이터 모델은 향후 다른 공급자를 지원할 수 있도록 유지할 수 있다.
- 별도 요청 전에는 Google 로그인, 계정 연결, 로그인 수단 관리, 계정 병합을 구현하지 않는다.

이전 문서에는 Google 로그인이나 Beta 1 계정 연결이 포함될 수 있다. 현재 승인된 구현 범위는 공급자 확장이 가능한 데이터 구조와 카카오 로그인만이다. 이전 범위를 구현하지 말고 충돌을 보고한다.

## 회원 탈퇴 규칙

- 탈퇴 요청 즉시 정상 계정 접근을 차단한다.
- 회원 상태를 `PENDING_DELETION`으로 변경한다.
- 7일 유예기간에는 승인된 최소 데이터만 보관한다.
- 삭제 대기 데이터는 정상 서비스나 AI 처리에 사용하지 않는다.
- 유예기간이 지나면 승인된 데이터를 영구 삭제한다.
- 유예기간 중 로그인해도 자동 복구하지 않는다.
- 복구에는 사용자의 명시적 확인이 필요하다.
- 복구가 성공적으로 완료되기 전에는 기존 기록을 노출하지 않는다.

## 인증 및 토큰 규칙

- Naroom은 카카오 로그인을 검증한 뒤 자체 Access Token과 Refresh Token을 발급한다.
- 카카오 Access Token을 Naroom 세션 토큰으로 사용하지 않는다.
- Naroom Refresh Token 원문을 저장하지 않는다.
- 승인된 보안 해시 또는 검증값과 최소 세션 메타데이터만 저장한다.
- 추후 카카오 API 호출이 필요한 별도 기능이 승인되지 않았다면 카카오 Access/Refresh Token을 저장하지 않는다.
- 구현 전 만료, 회전, 재사용 감지, 폐기, 로그아웃, 동시 세션 정책을 정의한다.
- 로컬 개발 편의를 위해 보안 기본값을 약화하지 않는다.
- 인가 코드, Access Token, Refresh Token, ID Token, 비밀값, 전체 Authorization 헤더를 로그에 남기지 않는다.
- 인증 테스트와 fixture는 가짜 값만 사용한다.

## 제품 불변 규칙

- 사용자의 기록은 AI 처리 성공 여부와 독립적으로 먼저 저장한다.
- AI 처리 실패로 원본 기록이 유실되면 안 된다.
- AI 결과를 확정적인 진단이나 판단으로 취급하지 않는다.
- 감정과 에너지 값은 사용자를 평가하는 점수가 아니라 관찰 정보다.
- 사용자 화면 용어는 `작은 실험`을 사용한다.
- 기존 내부 코드에서 확정됐다면 백엔드에서는 `challenge`를 사용할 수 있다.
- 별도 승인 없이 랭킹, 비교, 연속 기록 압박, 실패 중심 표현을 도입하지 않는다.

## 환경변수 및 비밀정보 규칙

환경변수 이름과 안전한 예시는 `.env.example`에서 관리한다.

실제 비밀값은 다음에 작성하지 않는다.

- 소스 코드
- 테스트 코드와 fixture
- 로그
- README와 프로젝트 문서
- `.env.example`
- OpenAPI 출력
- Git 커밋
- Claude 및 OMC 출력

Claude Code는 다음 내용을 읽거나 수정하거나 출력하지 않는다.

- `.env.local`
- `.env.*.local`
- DB 비밀번호
- JWT 서명 키
- OpenAI API 키
- 카카오 Client Secret
- Firebase 및 Google Cloud 서비스 계정 JSON
- 인증서와 개인키
- keystore
- 외부 서비스 토큰
- 실제 사용자 기록
- 실제 인증 토큰

코드나 Git diff에서 비밀값을 발견하면 값을 반복하지 않는다. 해당 파일과 노출 가능성만 보고한다.

## 수정 금지 및 주의 파일

명시적 요청 없이 다음을 수정하지 않는다.

- `build/`
- `.gradle/`
- `.idea/`
- `.env.local`
- `.env.*.local`
- `gradle/wrapper/gradle-wrapper.jar`
- Gradle Wrapper 버전과 구성
- 적용된 Flyway 마이그레이션
- `.gitignore`의 비밀정보 제외 규칙
- 생성된 OpenAPI 스냅샷의 수동 수정

Gradle Wrapper 변경이 필요하면 수정 전에 이유와 공식 Wrapper 명령을 제안한다.

## OMC 및 저장소 경계

- OMC 운영 규칙은 `.claude/CLAUDE.md`를 따른다.
- OMC 실행 상태를 프로젝트 기준 정보로 사용하지 않는다.
- 장기 보존할 결정은 승인된 프로젝트 문서에 남긴다.
- 백엔드 전용 작업에서 `naroom-app`을 수정하지 않는다.
- 해결되지 않은 동일 계약 결정을 병렬 Worker에게 중복 배정하지 않는다.
- 계약과 파일 소유권이 명확할 때만 병렬 구현한다.
- Worker가 승인된 API 계약을 독립적으로 변경하면 안 된다.
- 구현 중 계약 문제가 발견되면 계약을 바꾸기 전에 작업을 중단하고 보고한다.

## Git 작업 규칙

- `dev`는 일반 개발 및 검증 브랜치다.
- `main`은 안정 브랜치다.
- 별도 요청 없이 `feature/*` 브랜치를 만들지 않는다.
- Pull Request는 필수가 아니며 요청된 대규모 또는 고위험 변경에서만 사용한다.
- 명시적 요청 없이 커밋, push, merge, rebase, PR 생성, Issue 종료, 원격 설정 변경을 수행하지 않는다.
- 소스, 마이그레이션, 문서, 생성 계약, AI 도구 변경은 가능한 한 논리적으로 분리한다.
- 로컬 환경파일, OMC 실행 상태, 개인 설정, 비밀값, 사용자 데이터를 커밋하지 않는다.

## 작업 절차

1. `docs/PRODUCT_CONTEXT.md`를 읽는다.
2. 현재 Issue 또는 사용자 요구사항을 확인한다.
3. 관련 코드, 테스트, 마이그레이션, 계약을 확인한다.
4. 포함·제외 범위를 정의한다.
5. 수정 예정 파일과 검증 방법을 알린다.
6. 승인된 범위만 구현한다.
7. 집중 테스트, 전체 테스트, 빌드를 실행한다.
8. 공개 계약 변경 시 OpenAPI를 재생성한다.
9. 최종 diff와 비밀정보 포함 여부를 검토한다.
10. 변경사항, 명령, 결과, 위험, 미확정 사항을 보고한다.
