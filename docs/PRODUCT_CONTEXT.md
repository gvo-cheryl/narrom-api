# Naroom Product Context

## 문서 역할

이 문서는 AI 작업 도구와 개발자가 Naroom의 안정적인 제품 맥락을 이해하기 위한 구현용 요약 문서다.

- 상세 기획과 일정의 원본: Notion
- 코드 작업 범위와 완료 기준: GitHub Issue
- 제품의 장기적인 목적과 원칙: 이 문서
- 제품 방향이 변경되면 두 저장소의 이 문서를 함께 갱신한다.

현재 Issue의 요구사항과 이 문서가 충돌하면 임의로 구현하지 않고 사용자에게 확인한다.

## 제품 한 줄 정의

Naroom(나로움)은 기록과 AI를 통해 감정과 생각의 패턴을 이해하고, 사용자가 자신만의 기준으로 일상을 살아가도록 돕는 조용한 자기이해 플랫폼이다.

## 주요 사용자

주요 사용자는 생각이 많고 완벽주의, 자기비판, 번아웃을 자주 경험하는 25~39세 여성 직장인이다.

사용자는 다음 주제에 관심이 있을 수 있다.

- 감정과 일상 기록
- 자기이해
- 심리와 MBTI
- ADHD 특성 이해
- 작은 습관 형성
- 장기적인 변화와 패턴 발견

사용자는 기록의 필요성을 느끼더라도 꾸준히 기록하거나, 쌓인 기록에서 의미 있는 패턴을 발견하는 데 어려움을 겪을 수 있다.

## 해결하려는 문제

- 생각과 감정이 쌓여도 안전하게 정리할 공간과 방식이 부족하다.
- 기록을 남겨도 장기적인 감정과 행동 패턴을 발견하기 어렵다.
- 비교와 경쟁 중심의 서비스는 자기비판과 부담을 키울 수 있다.
- 완벽하게 기록해야 한다는 압박 때문에 기록을 포기할 수 있다.
- 사용자는 일방적인 평가보다 자신의 감정과 상황을 이해하도록 돕는 경험을 원한다.

## 핵심 가치

- 비교하지 않는다.
- 사용자를 평가하거나 재촉하지 않는다.
- 완벽한 기록보다 작은 기록의 지속을 돕는다.
- 기록하지 못한 날도 실패로 취급하지 않는다.
- AI는 정답을 단정하기보다 생각과 감정을 정리하도록 돕는다.
- 사용자가 자신만의 기준과 패턴을 발견하도록 돕는다.
- 조용하고 안전한 개인 공간을 지향한다.
- SNS형 공개 경쟁과 과도한 성취 압박을 지향하지 않는다.

## 핵심 경험

사용자는 Naroom에서 다음과 같은 경험을 할 수 있다.

1. 오늘의 감정과 생각을 부담 없이 기록한다.
2. AI의 도움을 받아 감정과 상황을 정리한다.
3. 작은 챌린지를 통해 가벼운 실천 동기를 얻는다.
4. 주간 및 월간 기록을 돌아본다.
5. 반복되는 감정과 행동 패턴을 발견한다.
6. 자신의 언어로 월간 자기요약을 작성한다.
7. 장기 기록을 통해 자신에게 맞는 기준과 생활 방식을 찾아간다.

## 주요 제품 영역

### Home

- 오늘의 상태 확인
- 기록 시작
- 진행 중인 작은 활동 확인
- 필요한 기능으로 빠르게 이동

### Records

- 일상, 감정, 생각 기록
- 기록 조회와 회고
- AI 기반 정리 및 분석
- 장기적인 패턴 확인

### Challenge

- 부담이 적은 작은 실천
- 진행 상황 확인
- 배지 등 가벼운 동기 요소

### Me

- 나의 기록과 요약
- 주간·월간·장기 분석
- 계정 및 앱 설정
- 개인정보와 계정 삭제

## 초기 제품 흐름

초기 제품에서 고려하는 주요 사용자 흐름은 다음과 같다.

- 회원가입 및 로그인
- 카카오 로그인
- 온보딩
- 일일 기록
- AI를 활용한 기록 정리
- 작은 챌린지
- 주간 리포트
- 나의 기록 확인
- 설정 및 계정 삭제
- 로딩, 오류, 빈 상태 및 취소 흐름

각 기능의 실제 구현 순서와 포함 범위는 현재 스프린트와 GitHub Issue를 기준으로 결정한다.

이 문서에 언급된 모든 기능을 한 번에 구현하지 않는다.

## MVP 원칙

- 초기 일정 안에 검증 가능한 범위에 집중한다.
- P0 기능은 가능한 한 적은 단계로 접근할 수 있도록 설계한다.
- 일정이 부족하면 핵심 기록과 자기이해 경험을 우선한다.
- 장기 분석, LifeTime, 월간 자기요약 등의 상세 범위는 스프린트 우선순위에 따라 조정한다.
- 기능을 임의로 확장하지 않는다.

## UX 및 카피 원칙

- 비판단적이고 차분한 표현을 사용한다.
- 따뜻하되 과도하게 감정적이거나 유아적인 표현은 피한다.
- 사용자의 감정을 단정하지 않는다.
- 의학적·정신건강 진단처럼 표현하지 않는다.
- “반드시”, “실패”, “부족”, “남보다”처럼 압박과 비교를 유발하는 표현을 피한다.
- 기록이 비어 있거나 중단된 상태를 부정적으로 평가하지 않는다.
- 사용자가 선택하고 건너뛸 수 있는 여지를 제공한다.

## AI 기능 원칙

- AI는 사용자의 감정과 생각을 정리하는 보조 도구다.
- AI의 답변을 진단, 치료 또는 절대적인 해석으로 표현하지 않는다.
- 사용자가 제공하지 않은 감정이나 의도를 사실처럼 단정하지 않는다.
- 사용자의 기록을 과도하게 일반화하지 않는다.
- 불확실한 해석은 가능성 또는 질문의 형태로 제시한다.
- 실제 사용자 기록을 개발 로그, 테스트 데이터 또는 AI 개발 대화에 노출하지 않는다.

## 데이터 및 보안 원칙

- 감정 기록과 자기서술은 민감한 개인 데이터로 취급한다.
- 필요한 데이터만 수집한다.
- 인증정보와 사용자 기록을 로그에 불필요하게 남기지 않는다.
- 실제 사용자 기록을 테스트 데이터로 사용하지 않는다.
- 사용자에게 계정과 데이터 삭제 흐름을 제공한다.
- API 키, 서비스 계정, 인증 토큰은 클라이언트에 포함하지 않는다.

## 저장소별 역할

### naroom-api

- 인증과 인가
- 기록 데이터 처리
- 챌린지 데이터 처리
- 리포트와 분석 처리
- 외부 AI 및 플랫폼 연동
- 모바일 앱을 위한 API 제공

### naroom-app

- iOS 및 Android 사용자 화면
- Expo Router 기반 탐색
- 기록 입력과 결과 표현
- 챌린지와 리포트 화면
- API 연동
- 로딩, 오류, 빈 상태와 접근성 처리

두 저장소는 별도 Git 저장소이며 모노레포를 전제로 하지 않는다.

## 기능 작업에 필요한 추가 정보

이 문서는 제품의 안정적인 큰 맥락만 제공한다.

각 기능을 시작할 때는 GitHub Issue 또는 작업 요청에 다음 내용을 별도로 명시한다.

- 기능을 만드는 이유
- 대상 사용자와 사용자 문제
- 포함 범위
- 제외 범위
- 화면과 사용자 흐름
- API 요청·응답 계약
- 오류, 빈 상태 및 취소 흐름
- 완료 기준
- 보안과 개인정보 영향
- 테스트 및 검증 방법
